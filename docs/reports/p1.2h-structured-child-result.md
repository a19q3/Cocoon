# P1.2h Structured Child Result Evidence

Last updated: 2026-05-19

## Summary

P1.2h hardens the P1.2 authority evidence path by adding structured child
results to the Redox authority children and fexeced fixture services.
Human-readable stdout markers remain in the logs, but the parent now parses the
structured result and binds that fact into the run/probe receipt body as:

```text
structured_child_result = true
```

This does not change the Redox launcher assumption, does not promote the final
`redox-enforced` label, and does not add supervisor, `contain`, dynamic FD
broker, bulk FD passing, or `pkgar` payload migration.

## What Changed

- `probe-authority` child emits a structured authority result after proving
  restricted namespace entry, allowed preopen read, denied path rejection, and
  hidden scheme rejection.
- `probe-fd-launch` emits a structured launcher result before `fexecve`; the
  controlled fixture emits a structured service result after FD launch.
- `probe-capsule-fd-launch` and `run --enforce-redox-authority` emit a
  structured launcher result before fexecing the installed entrypoint; the
  authority self-test service emits a structured service result after launch.
- Run and probe receipts record `structured_child_result`.
- `status`, `status --json`, and `audit` expose the structured-result evidence.

## Evidence Model

The current evidence model is:

```text
structured child result
+ child/command exit status
+ receipt body hash
+ receipt archive link
+ stdout/stderr log hash
+ status/log/audit readback
```

Stdout markers remain useful for human diagnostics and QEMU transcript review,
but they are no longer the primary parsed evidence source for P1.2h paths.
P1.2i adds local negative tests for malformed or incomplete structured evidence;
see [p1.2i-review-hardening.md](p1.2i-review-hardening.md).

## QEMU Evidence

`cargo xtask qemu-smoke` verifies that the structured evidence appears in:

- `probe-authority`
- `probe-fd-launch`
- `probe-capsule-fd-launch`
- `run --enforce-redox-authority`
- `status --json`
- `audit`
- `log-service`
- `network-denied-service`

The important PASS lines remain:

```text
PASS probe Redox authority inside redox
PASS probe Redox FD-only controlled service launch inside redox
PASS probe Redox FD-only installed capsule entrypoint inside redox
PASS cocoon run uses FD-only capsule entrypoint backend inside redox
PASS P1.2g log-service FD run profile inside redox
PASS P1.2g network-denied-service FD run profile inside redox
```

## Boundary

P1.2h is evidence hardening only. It keeps the intermediate authority mode:

```text
authority_mode = redox-enforced-capsule-entrypoint
authority_enforced_for_service = true
production_arbitrary_service = false
```

The final `redox-enforced` label remains blocked on review of the launcher
contract, broader service lifecycle semantics, and native packaging path.
