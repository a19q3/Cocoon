# P1.2f Redox FD Run Backend Evidence

Last updated: 2026-05-19

## Summary

P1.2f graduates the P1.2e installed capsule FD-only launch probe into an
explicit `cocoon run` backend:

```text
cocoon run <capsule-name> --enforce-redox-authority
```

The backend is Redox-only and reuses the same installed-entrypoint FD launch
path as `probe-capsule-fd-launch`. It opens the capsule entrypoint and declared
preopen evidence before restriction, enters a manifest-derived restricted
namespace, fexecs the installed entrypoint FD, captures stdout/stderr, and
writes a normal `capsule_run` receipt.

## What Is Proven

- `cocoon run --enforce-redox-authority` uses the same Redox FD launch backend
  as the installed-entrypoint probe.
- The run receipt records `authority_mode = redox-enforced-capsule-entrypoint`.
- The service boundary is marked with `authority_enforced_for_service = true`.
- `production_arbitrary_service = false` remains explicit.
- The run receipt records executable/preopen-before-restriction evidence,
  manifest-derived namespace entry, fexec success, declared resource read,
  denied ambient path rejection, and hidden `tcp` scheme rejection.
- The run receipt records `structured_child_result = true` when launcher and
  service evidence was parsed from structured child results.
- `status`, `logs`, and `audit` read back the enforced run receipt and captured
  stdout/stderr hashes.
- Non-Redox platforms fail closed with
  `Redox FD-only run backend unavailable on this platform`.

## What Is Not Proven

- Final production `authority_mode = redox-enforced`.
- Multiple installed service profiles under the same backend. This is covered
  by the follow-on P1.2g evidence report.
- Service supervision, restart, health checking, or crash recovery.
- `pkg/pkgar` payload integration.
- A generic sandbox CLI or `contain` backend.

## Receipt Mode

P1.2f intentionally uses the intermediate authority label:

```text
authority_mode = redox-enforced-capsule-entrypoint
authority_enforced_for_service = true
production_arbitrary_service = false
```

This distinguishes an audited installed-entrypoint backend from a final
production arbitrary-service claim. The final `redox-enforced` label should be
enabled only after multiple service profiles pass and the operational boundary
is reviewed.

## QEMU Evidence

`cargo xtask qemu-smoke` now includes:

```text
PASS probe Redox FD-only installed capsule entrypoint inside redox
PASS cocoon run uses FD-only capsule entrypoint backend inside redox
PASS audit Redox FD-only launch probe receipts inside redox
```

The run check verifies the run receipt output, `status` readback, stdout log
contents, and audit checks for:

- latest run receipt body hash
- latest run structured child result evidence
- latest run stdout/stderr log hash
- latest run FD launch fexec evidence
- latest run hidden scheme rejection evidence

## Harness Integrity

The QEMU harness treats Redoxer failures as hard failures for required P1.2f
commands. `verify`, `plan`, and the combined install/run/audit script use a
capturing runner that checks the process exit status and includes combined
stdout/stderr in the error if Redox exits non-zero. This prevents a failed
Redoxer invocation from being reported as a `TODO` smoke line.

The harness also stages a minimal Redoxer root at
`target/redox-smoke/redoxer-root` instead of mounting the whole repository into
`/root`. The staged tree contains only:

```text
bin/cocoon
capsules/hello-service.cocoon
capsules/hello-service-fd.cocoon
capsules/log-service.cocoon
capsules/network-denied-service.cocoon
capsules/hello-service-v2.cocoon
```

Inside Redox, the harness uses `/root/redoxer-root/...` paths. This avoids
copying host build caches or unrelated repository state into the QEMU image and
keeps the smoke input auditable.

## Compatibility Note

P1.2f adds FD-launch evidence fields to `capsule_run` receipts. Audit accepts
both the current receipt body hash and the legacy pre-P1.2f run receipt body
shape when all new FD-launch fields are absent. This preserves existing receipt
chains while still requiring complete FD-launch evidence whenever
`authority_enforced_for_service = true`.

## Follow-On

P1.2g adds two more Redox service profiles under the same backend. The default
Redox `run` behavior and final `redox-enforced` production label remain
reserved for later operational review.
