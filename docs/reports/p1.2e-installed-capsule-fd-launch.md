# P1.2e Installed Capsule FD-Only Launch Evidence

Date: 2026-05-17

## Summary

P1.2e proves Cocoon can launch an installed capsule entrypoint from an
already-open executable file descriptor under a manifest-derived restricted
Redox namespace. This graduates P1.2d from a controlled internal fixture to a
real materialized capsule payload entrypoint.

This is still a probe, not production `cocoon run` enforcement. Production
`run` remains fail-closed unless `--allow-unenforced-authority` is passed.

## What Is Proven

- Cocoon resolves `entry.cmd` from the installed manifest into the materialized
  capsule payload tree.
- Cocoon opens the installed capsule entrypoint before namespace restriction.
- Cocoon opens declared preopen evidence before namespace restriction.
- The child enters a manifest-derived restricted Redox namespace containing the
  runtime minimum schemes plus declared Redox runtime schemes.
- Cocoon launches the installed entrypoint through inherited executable FD using
  Redox `fexecve`.
- The launched service reads a declared resource through an inherited preopen FD.
- Denied ambient file path access fails.
- Undeclared `tcp` scheme access fails.
- Cocoon records a `capsule_fd_launch_probe` receipt.
- `cocoon audit` verifies the probe receipt body, archive link, and captured
  stdout/stderr hashes.
- QEMU smoke accepts:
  `PASS probe Redox FD-only installed capsule entrypoint inside redox`.
  It also accepts:
  `PASS audit Redox FD-only launch probe receipts inside redox`.

## What Is Not Proven

- Production `cocoon run` authority enforcement.
- `authority_mode = redox-enforced` for arbitrary production services.
- Multiple independent service profiles under the same launcher boundary.
- Service supervision, restart, health checks, or reboot recovery.
- `pkg`/`pkgar` payload integration.
- Generic sandbox UX or a `contain` backend.

## Authority Mode

P1.2e records installed-entrypoint evidence as:

```text
authority_mode = redox-enforced-capsule-entrypoint
authority_enforced_for_service = true
production_arbitrary_service = false
```

The label is deliberately narrower than final production enforcement. It proves
the installed payload entrypoint can cross the FD-only boundary, but Cocoon
should only promote `redox-enforced` after multiple service profiles use this
path through `cocoon run`.

## Validation

The Redox/QEMU acceptance output includes:

```text
PASS probe Redox authority inside redox
PASS classify Redox FD-only service launch gap inside redox
PASS probe Redox FD-only controlled service launch inside redox
PASS probe Redox FD-only installed capsule entrypoint inside redox
PASS reject unenforced authority run inside redox
```

## Next Stage

P1.2f should turn the installed-entrypoint probe into an internal `run` backend
for selected service profiles while keeping the public production label
conservative:

```text
open installed entrypoint/preopens before restriction
enter manifest-derived restricted namespace
fexec entrypoint FD
capture stdout/stderr and run receipt
record redox-enforced-capsule-entrypoint until multiple profiles pass
```

Do not add supervisor, `contain`, or `pkgar` work in this phase.
