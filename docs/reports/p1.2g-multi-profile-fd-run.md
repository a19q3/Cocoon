# P1.2g Redox FD Run Multi-Profile Evidence

Last updated: 2026-05-18

## Summary

P1.2g extends the P1.2f Redox FD-only run backend from one installed capsule
entrypoint fixture to multiple service profiles:

```text
hello-service-fd
log-service
network-denied-service
```

Each profile is installed as its own capsule, opened before restriction, run
with `cocoon run --enforce-redox-authority`, and audited through the normal
`capsule_run` receipt path.

## What Is Proven

- The explicit Redox FD run backend is reusable across multiple installed
  service profiles.
- Each profile writes a normal run receipt with
  `authority_mode = redox-enforced-capsule-entrypoint`.
- Each profile records executable/preopen-before-restriction evidence,
  manifest-derived namespace entry, fexec success, declared resource read,
  denied ambient path rejection, and hidden `tcp` scheme rejection.
- `status`, `logs`, and `audit` read back the enforced run receipts and
  captured stdout/stderr hashes for the extra profiles.
- QEMU smoke asserts profile markers from service stdout so the test proves the
  additional installed entrypoints actually executed under the backend.

## What Is Not Proven

- Final production `authority_mode = redox-enforced`.
- Long-running service supervision or restart semantics.
- `pkg/pkgar` payload integration.
- A generic sandbox CLI or `contain` backend.

## QEMU Evidence

`cargo xtask qemu-smoke` now includes:

```text
PASS cocoon run uses FD-only capsule entrypoint backend inside redox
PASS P1.2g log-service FD run profile inside redox
PASS P1.2g network-denied-service FD run profile inside redox
```

The additional profile checks require:

- `Ran <profile>@0.1.0`
- `Authority mode: redox-enforced-capsule-entrypoint`
- `Latest run authority enforced for service: true`
- `Audit passed for <profile>`
- `latest run FD launch fexec: true`
- `latest run FD launch hidden scheme: true (/scheme/tcp)`
- profile-specific stdout marker from the fexeced service

## Boundary

P1.2g strengthens confidence in the Redox FD-only run backend, but it still uses
the intermediate label:

```text
authority_mode = redox-enforced-capsule-entrypoint
authority_enforced_for_service = true
production_arbitrary_service = false
```

The final `redox-enforced` production label remains reserved for a reviewed
operational boundary and broader service lifecycle work.
