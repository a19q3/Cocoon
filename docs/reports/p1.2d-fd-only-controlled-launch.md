# P1.2d FD-Only Controlled Launch Evidence

Date: 2026-05-17

## Summary

P1.2d proves a controlled Redox service can be launched from an already-open
executable file descriptor under a restricted Redox namespace. This is the first
Cocoon evidence that moves beyond a null-namespace probe into actual process
execution through Redox FD authority.

The probe is intentionally not wired into production `cocoon run`. Production
run remains fail-closed unless `--allow-unenforced-authority` is passed.

## What Is Proven

- Cocoon opens the executable before namespace restriction.
- Cocoon opens declared preopen evidence before namespace restriction.
- Cocoon enters a restricted Redox namespace in the child process.
- Cocoon can launch a controlled fixture through inherited executable FD using
  Redox `fexecve`.
- The controlled fixture can read the declared preopen FD after launch.
- Denied ambient file path access fails after launch.
- Hidden/undeclared `tcp` scheme access fails after launch.
- Cocoon records an `fd_launch_probe` receipt.
- QEMU smoke accepts the controlled-service enforcement line:
  `PASS probe Redox FD-only controlled service launch inside redox`.

## What Is Not Proven

- Arbitrary installed capsule entrypoint launch.
- Production `cocoon run` authority enforcement.
- `authority_mode = redox-enforced` for arbitrary services.
- Service supervision, restart, health, or reboot recovery.
- `pkg`/`pkgar` payload integration.
- Generic sandbox UX or a `contain` backend.
- Dynamic launcher behavior for multiple real service profiles.

## Authority Mode

P1.2d records controlled-service evidence as:

```text
authority_mode = redox-controlled-service-enforced
authority_enforced_for_service = true
production_arbitrary_service = false
```

This distinction is deliberate. The controlled fixture proves the FD-only launch
mechanism and restricted namespace behavior, but it is not yet arbitrary capsule
execution.

## Validation

The implementation is covered by these commands:

```bash
cargo fmt --all --check
cargo check -p cocoon-runtime
cargo check -p cocoon-cli
cargo check -p cocoon-cli --target x86_64-unknown-redox
cargo clippy --all-targets --all-features -- -D warnings
cargo test -p cocoon-runtime
cargo test -p cocoon-cli
cargo xtask test
cargo xtask redox-smoke
cargo xtask qemu-smoke
cargo xtask redox-package
```

The Redox/QEMU acceptance output includes:

```text
PASS probe Redox authority inside redox
PASS classify Redox FD-only service launch gap inside redox
PASS probe Redox FD-only controlled service launch inside redox
PASS reject unenforced authority run inside redox
```

## Next Stage

P1.2e should graduate from controlled fixture launch to installed capsule
entrypoint launch:

```text
open installed capsule entrypoint before restriction
open declared preopens before restriction
enter manifest-derived restricted namespace
fexec installed service entrypoint
prove declared resource access
prove denied ambient path and undeclared scheme rejection
record redox-enforced-capsule-entrypoint evidence
```

Do not promote the final `redox-enforced` production label until multiple
arbitrary service profiles pass through the same boundary.
